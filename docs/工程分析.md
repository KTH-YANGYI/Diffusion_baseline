# 工程分析

## 1. 这个仓库现在是什么结构

仓库已经从“按某一条方法线组织”改成“按任务域组织”。

当前根目录的角色是：

- `data/`
  原始或整理后的任务数据
- `artifacts/`
  manifest、ROI、运行结果、历史归档
- `configs/`
  各方法线配置
- `src/`
  代码实现
- `docs/`
  说明文档

这意味着后面即使再加别的 line，也不需要再把整个仓库建在某个方法名下面。

## 2. 当前已有的方法线

目前已接入的是：

- `src/crack_synth/methods/diffusion_baseline/`

它的定位是：

- 用真实裂缝样本生成 donor ROI
- 用同视频正常背景做背景 patch
- 用预训练 `Stable Diffusion Inpainting` 做局部裂缝生成

它不是：

- 从零训练 diffusion
- 当前不含 ControlNet
- 当前不含 LoRA

## 3. 新目录为什么比旧目录更稳

旧结构的问题是：

- `dataset_new/`、`manifests/`、`outputs/`、`baseline/` 全在根目录并列
- 路径和方法线强耦合
- 后面再加别的方法线时，产物和配置容易混

现在的结构把这几个层次拆开了：

- 数据放 `data/contact_wire_v1/`
- manifest 放 `artifacts/contact_wire_v1/manifests/`
- 当前方法线输出放 `artifacts/contact_wire_v1/methods/diffusion_baseline/`
- 历史服务器拷回结果放 `artifacts/contact_wire_v1/archives/server_exports/`

所以后面再加：

- `controlnet`
- `copy_paste`
- `lora_finetune`

都可以直接挂在 `methods/` 下面，不会和这条线互相污染。

## 4. 当前配置怎么驱动路径

这次重构最关键的一点是：

**代码不再默认依赖根目录下固定名字的 `dataset_new / manifests / outputs`。**

当前配置文件：

```text
configs/methods/diffusion_baseline/contact_wire_v1.yaml
```

里面显式声明了：

- `repo_root`
- `dataset_root`
- `manifests_root`
- `output_root`

所以你后面如果要换数据线或者换方法线，只需要新增配置，不需要再改一批硬编码路径。

## 5. 当前 diffusion baseline 的真实流程

主流程还是三步：

```text
build_manifests
    -> prepare_rois
        -> generate_baseline
```

但现在三步对应的产物位置已经变成：

- manifest:
  `artifacts/contact_wire_v1/manifests/`
- ROI 与生成结果:
  `artifacts/contact_wire_v1/methods/diffusion_baseline/`

## 6. donor 在这套系统里的真实角色

`donor` 不是模型的一部分，而是：

**真实裂缝样本导出的几何模板资产。**

它主要贡献：

- 裂缝大致位置
- 裂缝长度和形状
- 裂缝的编辑区域

当前真正送进模型的不是 donor 原图，而是：

- 正常背景 patch
- `mask_edit_roi`
- `prompt`
- `negative_prompt`

## 6.1 当前已经有独立的生成后评估入口

现在除了生成本身，仓库里也增加了单独的生成质量评估入口：

- `evaluate_generation`

它会对某个 `run_dir` 输出每张图的质量指标和整体汇总，重点衡量：

- 改动是否集中在编辑区
- 编辑区外背景是否被过度改坏
- raw mask 区域的裂缝对比和边缘是否相对背景变强

## 7. 当前背景对齐机制

这条 baseline 现在的关键假设仍然没有变：

- donor 先记录一个 `crop_box_xyxy`
- 正常背景只从同 `video_id` 里挑
- 优先选 `frame_id` 接近的背景
- 用同一个 `crop_box_xyxy` 去裁背景

所以它是：

**“同视频 + 近邻帧 + 同裁剪框迁移”的 baseline**

不是显式图像配准。

这做法合理，但不是严格几何对齐。

## 8. 当前代码为了兼容旧产物做了什么

重构后，旧 manifest 或 donor manifest 里可能还残留旧相对路径，比如：

- `dataset_new/...`
- `manifests/...`
- `outputs/baseline/...`

这次代码里加了兼容路径重写逻辑，会优先把这些旧前缀自动映射到新结构下的：

- `data/contact_wire_v1/...`
- `artifacts/contact_wire_v1/manifests/...`
- `artifacts/contact_wire_v1/methods/diffusion_baseline/...`

所以你现在既可以继续重跑，也不至于因为历史相对路径马上全失效。

## 9. 我对现在这个架构的判断

这次重构后的结构更适合你后面继续做多条线实验。

它的优点是：

- 任务数据和方法实现分开
- 产物和代码分开
- 各方法线可以共用同一套数据与评估口径
- 上传服务器时不再需要解释一堆“这个顶层目录到底是什么”

如果后面继续扩展，我建议遵守两个原则：

1. 每新增一条 line，都放到 `src/crack_synth/methods/<method_name>/`
2. 每新增一条 line，都配一份 `configs/methods/<method_name>/<dataset_id>.yaml`
