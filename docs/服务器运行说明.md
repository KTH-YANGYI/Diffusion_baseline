# 服务器运行说明

这份文档只讲一件事：

**把当前仓库上传到服务器后，怎样把 `diffusion_baseline` 跑起来。**

## 1. 先确认服务器环境

当前项目的 Python 版本要求是：

```text
>=3.10, <3.12
```

也就是说，服务器上建议直接使用：

- Python 3.10

不要用：

- Python 3.12
- Python 3.13
- Python 3.14

如果你按仓库里的 `environment.yml` 建环境，会自动使用 Python 3.10。

## 2. 上传哪些目录

如果你是要在服务器上重新跑一遍，最少上传这些内容就够了：

- `configs/`
- `data/`
- `docs/`
- `src/`
- `pyproject.toml`
- `environment.yml`
- `README.md`

`artifacts/` 不是必须上传。

原因是：

- `build_manifests` 会重新生成 manifest
- `prepare_rois` 会重新生成 donor ROI
- `generate_baseline` 会重新生成输出

只有在下面两种情况下，才建议一起上传 `artifacts/`：

- 你想保留本地已经跑出来的中间产物
- 你想在服务器上继续接着已有结果往下跑

你现在本地已经把 manifest 和 baseline 相关产物删掉了，所以按当前状态：

- 直接不上传 `artifacts/`
- 让服务器从零生成新的 `artifacts/contact_wire_v1/...`

就是最合适的做法。

## 3. 上传后进入仓库根目录

```bash
cd /your/path/crack-synth
```

下面所有命令都默认在这个目录下执行。

## 4. 创建环境

```bash
conda env create -f environment.yml
conda activate crack-synth
```

## 5. 安装 PyTorch

根据服务器 CUDA 版本安装对应的 PyTorch。

例如 CUDA 12.1：

```bash
pip install torch --index-url https://download.pytorch.org/whl/cu121
```

如果只是做 CPU 检查：

```bash
pip install torch --index-url https://download.pytorch.org/whl/cpu
```

## 6. 安装当前项目

```bash
pip install -e .
```

安装完成后，这三个命令就应该可用：

- `build_manifests`
- `prepare_rois`
- `generate_baseline`

## 7. 先做一次环境检查

建议先确认 PyTorch 和 CUDA 是否正常：

```bash
python -c "import torch; print(torch.__version__); print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'cpu')"
```

再确认项目命令已经装好：

```bash
build_manifests --help
prepare_rois --help
generate_baseline --help
```

## 8. 当前默认配置

当前默认配置文件是：

```text
configs/methods/diffusion_baseline/contact_wire_v1.yaml
```

这个配置里已经显式定义了：

- `dataset_root = data/contact_wire_v1`
- `manifests_root = artifacts/contact_wire_v1/manifests`
- `output_root = artifacts/contact_wire_v1/methods/diffusion_baseline`

如果这些目录当前不存在，也没关系：

- `build_manifests` 会创建 `artifacts/contact_wire_v1/manifests/`
- `prepare_rois` / `generate_baseline` 会创建 `artifacts/contact_wire_v1/methods/diffusion_baseline/`

所以现在代码不再依赖旧结构里那种固定顶层目录：

- `dataset_new/`
- `manifests/`
- `outputs/baseline/`

## 9. 第一步：重建 manifest

```bash
build_manifests --config configs/methods/diffusion_baseline/contact_wire_v1.yaml
```

输出位置：

```text
artifacts/contact_wire_v1/manifests/
```

如果服务器上这个目录一开始不存在，这是正常的；命令会自动创建。

建议先看：

```bash
python -c "import json; print(json.load(open('artifacts/contact_wire_v1/manifests/split_summary.json', encoding='utf-8'))['folds'][0])"
```

## 10. 第二步：准备 donor ROI

```bash
prepare_rois --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml
```

如果先做 smoke test：

```bash
prepare_rois --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --max-samples 5
```

输出位置：

```text
artifacts/contact_wire_v1/methods/diffusion_baseline/fold_0/roi_assets/
```

如果 `artifacts/contact_wire_v1/methods/diffusion_baseline/` 还不存在，这一步也会自动创建。

这里几个概念要分清：

- `donor`
  指真实裂缝样本导出的 ROI 资产
- `mask_raw_roi`
  精确裂缝标签
- `mask_edit_roi`
  给 inpainting 模型的编辑区域

## 11. 第三步：先做计划，不直接推理

```bash
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --plan-only
```

小规模检查：

```bash
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --plan-only --max-donors 12 --max-backgrounds 2 --max-seeds 2
```

## 12. 第四步：正式推理

```bash
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml
```

如果先做一轮小样本测速：

```bash
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --max-donors 5 --max-backgrounds 2 --max-seeds 2
```

当前真正送进模型的输入是：

- 正常背景 patch
- `mask_edit_roi`
- `prompt`
- `negative_prompt`

当前 donor 原图没有直接作为视觉条件输入模型。

## 13. 当前背景对齐逻辑

当前版本不是严格图像配准。

它的做法是：

1. donor 侧先记录 `crop_box_xyxy`
2. 在同一 `video_id` 的正常背景中选图
3. 优先选 `frame_id` 接近的背景
4. 用同一个 `crop_box_xyxy` 直接裁背景
5. 在这个背景 patch 上做 inpainting

所以它依赖一个 baseline 假设：

**同视频近邻帧的局部位置足够接近。**

正式大规模推理前，建议先抽样检查若干 donor/background 配对是否自然。

## 14. 如果服务器不能联网下载模型

把配置中的：

```yaml
model_id_or_path: stable-diffusion-v1-5/stable-diffusion-inpainting
```

改成本地路径，例如：

```yaml
model_id_or_path: /data/models/sd15-inpaint
local_files_only: true
```

## 15. 推荐的最稳运行顺序

这套顺序默认你上传的是“干净仓库”，也就是没有现成 manifest、ROI 和 baseline 输出：

```bash
conda env create -f environment.yml
conda activate crack-synth
pip install torch --index-url https://download.pytorch.org/whl/cu121
pip install -e .
build_manifests --config configs/methods/diffusion_baseline/contact_wire_v1.yaml
prepare_rois --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --max-samples 12
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --plan-only --max-donors 12 --max-backgrounds 2 --max-seeds 2
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --max-donors 5 --max-backgrounds 2 --max-seeds 2
```

也就是说，服务器端会按下面顺序重新创建产物：

1. `build_manifests` 重新生成 manifest
2. `prepare_rois` 重新生成 donor ROI
3. `generate_baseline` 重新生成 plan 和推理结果

如果这些都没问题，再去掉限制参数跑完整第 0 折：

```bash
prepare_rois --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml --plan-only
generate_baseline --fold 0 --config configs/methods/diffusion_baseline/contact_wire_v1.yaml
```

## 16. 最应该先检查哪些文件

- `artifacts/contact_wire_v1/manifests/split_summary.json`
- `artifacts/contact_wire_v1/methods/diffusion_baseline/fold_0/roi_assets/donor_manifest.jsonl`
- `artifacts/contact_wire_v1/methods/diffusion_baseline/fold_0/run_<timestamp>/summary.json`
- `artifacts/contact_wire_v1/methods/diffusion_baseline/fold_0/run_<timestamp>/planned_pairs.jsonl`

这几个文件基本就能告诉你：

- manifest 是否生成对了
- donor ROI 是否生成对了
- 背景配对是否合理
- 哪些样本被跳过了
